# 终端易用性与用户教程设计

## 目标

在不改变注册流水线、认证流水线、限流策略和凭据格式的前提下，为注册端与本地认证端建立一致的终端交互。默认输出面向日常使用，只展示用户需要采取行动或能够判断运行状态的信息；Debug 输出保留现有队列、阶段、耗时、重试和冷却探测等诊断数据。

本设计只涉及命令行终端，不引入网页、桌面界面或全屏 TUI。输出采用事件追加方式，只有状态发生变化或用户显式命令需要返回结果时才新增一行，不使用固定周期刷屏，也不原地覆盖历史记录。

## 范围

本次覆盖两个入口：

- `start.sh` 启动的账号注册程序；
- `auth-service.sh` 启动的本地常驻认证服务。

同时整理面向用户的中文教程，包括首次运行、注册配置、认证服务配置、运行状态解读、账号取用和常见故障。注册、认证、同步、凭据库存和 CPA 文件格式的业务行为不在本次修改范围内。

## 交互模式

### 普通模式

普通模式是两个入口的默认模式。它只输出以下事件：

1. 启动检查进入新阶段或完成；
2. 服务开始、暂停、恢复、停止；
3. 有新任务开始；
4. 任务成功、可恢复失败或终止失败；
5. 进入限流、执行恢复探测、解除限流；
6. 远端来源首次连接、连接状态改变或发现新账号；
7. 账号库存取用成功或失败；
8. 用户显式请求状态。

静默轮询、无变化的快照同步、队列内部搬运和周期性性能采样不产生普通输出。持续运行的服务没有确定的总任务数，因此进度使用累计完成数、当前待处理数、本次运行平均速率和库存数量表达，不使用误导性的百分比进度条。

建议格式如下：

```text
[✓] 认证服务已启动 | 待处理 64 | 可用凭据 1206
[→] 开始认证 #1207 | 待处理 63
[✓] 认证成功 #1207 | 运行平均 5.7/分 | 可用凭据 1207
[⏸] 触发限流 | 60秒后单次探测
[▶] 限流解除 | 实际等待 61秒
```

任务编号是本次进程内的授权或注册启动序号，从 1 开始，重启后重新计数且不持久化。它只用于关联本次运行的开始与结果事件，不等同于累计成功数；累计成功数由持久化记录单独显示。不显示邮箱、SSO、Cookie、OAuth token、SSH 主机或文件中的凭据内容。失败信息使用稳定、可操作的中文分类；仅在 Debug 模式附带底层原因码。

认证端“待处理”定义为当前有效原子快照内尚无 `imported` ledger 记录的唯一账号数，包含已排队、已准备、正在认证和等待重试的账号，而不是容量固定的内存队列深度。快照尚不可用时显示 `—`，不得用 `64` 等队列容量冒充总待处理数。注册端仅在设置 `TARGET` 时显示 `max(TARGET - 累计成功数, 0)`；无限运行时显示“持续运行”，不显示待处理数量。

“运行平均速率”以本次进程内已经原子落盘并结算成功的数量为分子，以进程启动后的完整运行时间为分母；启动准备、正常处理和全部限流冷却时间都计入分母。尚无一次成功时显示 `—`，不使用持久化历史累计数推算本次速度。近五分钟滑动速率保留为 Debug 诊断指标，不进入普通模式。

普通模式下的 `s` 命令返回一行摘要：运行状态、待处理数量、当前阶段、本次运行平均速率、累计成功、可用和已取用凭据，以及限流状态。它不输出三段内部队列深度、探测锁、节拍剩余时间等实现细节。

### Debug 模式

Debug 模式保留现有诊断能力，包括：

- 注册端的 T/Q 库存、物理并发、S/P/C 阶段耗时和 token 求解时间；
- 认证端的 source/prepared/completion 队列、active stage、retry waiting、pace、cooldown、probe、attempt success 和 eventual success；
- 稳定的底层错误原因码和必要的异常摘要。

Debug 模式允许注册端继续按现有采样周期输出性能面板。认证端仍以事件输出为主；用户执行 `s` 时输出完整诊断快照。任何模式都不得输出凭据或可直接识别账号的信息。

两个入口统一支持 `--debug`。现有 `REGISTER_LOG_MODE=debug` 继续兼容；认证端使用 `XAI_AUTH_SERVICE_LOG_MODE=user|debug` 作为非交互部署入口。命令行参数优先于环境变量。普通模式不要求用户设置任何变量；非法模式值在启动前报配置错误，而不是静默回退。

## 终端组件边界

业务流水线继续产生结构化事件，终端渲染器负责选择、翻译和格式化事件。业务层不得拼接面向用户的完整句子，渲染器也不得改变调度、重试、限流或持久化状态。

注册端沿用现有用户事件和 Debug 指标快照，只统一事件命名、字段顺序和启动提示。认证端将当前 `EventTerminal` 分为两层：

- 结构化事件到用户状态模型的归类；
- 按 `user` 或 `debug` 模式渲染文本。

两个入口共享一组轻量终端约定，而不强制共享业务事件类型。约定包括符号、状态词、速率精度、计数格式和隐私字段过滤。这样可以保持视觉一致，同时避免把两套不同流水线耦合到一个庞大的公共抽象。

## 状态与数据流

普通输出的数据流为：

```text
注册/认证流水线状态变化
        -> 结构化事件
        -> 模式过滤与用户字段投影
        -> 单行事件文本
        -> stdout
```

Debug 模式在同一事件流上增加诊断字段，并保留注册端的周期性指标快照。终端渲染失败不得中断注册或认证；未知事件在普通模式忽略，在 Debug 模式输出不含敏感字段的事件类别和错误摘要。

认证端渲染器维护极少量显示状态，用于抑制重复的“来源已连接”和无新增账号的快照更新。它不维护业务真相；所有计数和限流状态仍来自 pipeline、ledger 和 inventory。

## 启动与控制

`start.sh` 首次运行继续完成依赖安装和邮箱模式配置，但启动阶段采用固定编号或明确阶段名，让用户知道当前处于系统依赖、Python 环境、浏览器依赖、配置还是正式运行阶段。已有配置直接启动时不重复展示首次安装教程。

两个入口默认在当前终端前台运行，停止后重复执行同一命令即为重启。核心使用流程不得要求 `tmux`、`screen`、systemd、launchd 或其他会话管理、守护进程依赖。

`auth-service.sh` 启动后先显示一段短摘要：服务用途、来源连接状态、输出目录的逻辑名称和可用控制命令。控制提示保持一行：`s` 状态、`take N` 取用、`p` 暂停、`r` 恢复、`c` 取消当前任务、`q` 退出。路径可以显示用户选择的本地输出目录，但不得显示 SSH 密钥路径、主机、账号标识或远端凭据来源内容。

交互终端使用标准库维护固定的 `认证> ` 输入行，不增加第三方终端依赖。业务事件到达时先清除当前显示行，输出事件后恢复尚未提交的输入缓冲；`take N` 等命令因此始终可发现且不会被滚动日志吞没。非 TTY 输入继续使用逐行模式，不输出 ANSI 控制序列。

普通和 Debug 模式使用相同控制命令。暂停、恢复、取消和退出的语义保持不变。

## 错误处理

错误按用户可操作性分为四类：

- 配置错误：启动前终止，并指出缺少的配置名和对应教程章节；
- 暂时性失败：说明将自动重试，不要求用户操作；
- 限流：显示当前处于冷却以及下次单次探测时间；
- 终止性失败：说明该任务已跳过或服务已停止，并给出 Debug 启动方式。

普通模式不展示 traceback。Debug 模式可以展示异常类别和经过净化的摘要；原始凭据、Cookie、邮箱、授权码和密钥始终禁止进入终端。现有流水线的失败分类和重试政策不因显示层而改变。

## 教程结构

README 保留最短可执行路径和项目概览，并增加“用户教程”入口，避免继续堆叠所有细节。详细内容拆分为以下中文教程：

1. `docs/guides/registration.md`：安装、邮箱模式、性能参数、启动与停止；
2. `docs/guides/auth-service.md`：远端同步、本地目录、前台运行和控制命令；
3. `docs/guides/credential-inventory.md`：`available`、`claiming`、`claimed` 的含义，`take N` 的行为和备注字段；
4. `docs/guides/runtime-troubleshooting.md`：普通事件解释、Debug 字段、限流与来源断开。

所有命令使用占位主机、占位路径和占位密钥，不写入实际服务器、域名、邮箱、token 或本机私密路径。教程以 `bash start.sh` 和 `bash auth-service.sh` 为主入口；底层 Python 命令只放在开发或排障章节。

## 验证

本次不新增与业务无关的大型测试集合。复用并调整现有终端输出测试，只覆盖容易回归且风险较高的契约：

1. 普通模式不会周期刷出 Debug 面板；
2. 普通模式在成功、失败、限流、状态变化和 `s` 命令时输出预期摘要；
3. Debug 模式保留现有队列、阶段、节拍与冷却字段；
4. `--debug` 和兼容环境变量的优先级正确；
5. 两种模式都不会输出测试记录中的账号标识或凭据；
6. 模式切换不改变任务结果、重试、限流、同步、库存和退出行为。

教程通过命令、文件名、参数和锚点检查验证。最终执行现有注册运行时测试、认证服务测试和完整测试集，并用模拟事件人工检查普通与 Debug 两种终端输出。正在运行的认证服务不用于试验未验证的显示层代码；新实现通过验证后再在可控重启时应用。

## 完成标准

- 新用户只运行入口脚本即可看到清晰的启动阶段、当前工作和结果；
- 日常运行无周期性调试刷屏，所有新增行都对应用户可感知的状态变化或显式命令响应；
- 普通状态能回答“是否在运行、有没有成功、当前速度、是否限流、还有多少待处理、凭据有多少”；
- Debug 模式完整保留现有诊断信息；
- 注册端与认证端使用一致的符号、术语和参数；
- README 和分篇教程中的步骤可以直接执行且不包含隐私信息；
- 现有注册、认证、限流、库存和 CPA 兼容行为保持不变。
